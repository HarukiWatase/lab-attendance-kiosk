# 運用・設定ガイド（フェーズ別）

本書は、現状のシステム（FastAPI / フロント / GAS / スプレッドシート）を**目的に応じて段階的に使う**ための整理資料です。

- **今すぐのゴール（フェーズ1）**: 打刻が `attendance_log` に問題なく蓄積されること。
- **分析ダッシュボード・週次集計・フロントの分析強化**は**後フェーズ**とし、フェーズ1では必須にしません。

---

## 0. 全体像（短い参照）

| コンポーネント | 役割 |
|----------------|------|
| フロント | QR相当の `user_id` 入力 → バックエンドへ POST |
| FastAPI | 入力検証・クールダウン・（本番では）`user_master` 照合 → GAS Web アプリへ POST |
| GAS（Web アプリ） | `doPost` でスプレッドシート `attendance_log` に1行追加 |
| スプレッドシート | 生ログの保管場所（ほかシートは分析フェーズで活用） |

**GAS の「定期実行」**が指すのは、現状コードでは **`runAutoFixBatch` を毎日 5時（スクリプトのタイムゾーン基準）に動かすトリガー**だけです。打刻のたびに動くのは **`doPost`（Web アプリの URL への POST）**であり、トリガーとは別物です。

## フェーズ1 — 記録の蓄積だけを安定させる（最優先）

### 1-1. スプレッドシートで用意するもの

| シート名 | フェーズ1で必須か | 内容 |
|----------|-------------------|------|
| `attendance_log` | 実質必須（自動生成可） | ヘッダ例: `timestamp`, `user_id`, `action`, `source`, `request_id`, `note`。初回 `doPost` 時に GAS が無ければ作成します。 |
| `user_master` | **本番運用では必須** | ヘッダ例: `user_id`, `display_name`, `active`, …。FastAPI（`APP_ENV` が本番相当で GAS URL が有効なとき）は、`doGet` の `mode=users` 経由でここを読み、**未登録・無効ユーザーは打刻拒否**します。 |

**開発だけ**（`APP_ENV=dev` かつ `GAS_WEBHOOK_URL` が未設定またはプレースホルダ）なら、バックエンドはローカルモックユーザで動き、`user_master` なしでも打刻の動作確認ができます。

### 1-2. GAS（Apps Script）で必ず行うこと

1. 本リポジトリの `gas/attendance.gs` と同等のスクリプトをスプレッドシートに紐づけて保存する。
2. **スクリプトプロパティ**に共有秘密を入れる（`setScriptProperties()` をエディタで1回実行するか、プロジェクトの設定から手入力）:
   - `GAS_SHARED_SECRET` … バックエンド `.env` の `GAS_SHARED_SECRET` と**同一**の文字列。
   - `AUTO_FIX_HOURS` … 仕様上 `0` でよい（自動補正で「退勤時刻」を何時間ずらすか。`0` は前日未退勤者に対し**同日時刻の補正退勤**を書く動きになる）。
3. **デプロイ → ウェブアプリ**を作成し、**POST を受ける URL** を控える。
   - **実行ユーザー**: スプレッドシートを編集できるアカウント。
   - **アクセス**: バックエンドから到達できる設定（通常は「全員」または組織内）。
4. バックエンド `.env` に `GAS_WEBHOOK_URL`（上記 URL）と `GAS_SHARED_SECRET` を設定する。

### 1-3. 定期実行トリガー（`installDailyBatchTrigger`）— フェーズ1では任意

| 項目 | 内容 |
|------|------|
| 何をするか | `runAutoFixBatch` を **1日1回**（コード上は「毎日 5 時」）実行するトリガーを追加する。 |
| 処理内容 | **前日**の `attendance_log` を走査し、「最後の行動が `出勤` のまま `退勤` が無い」ユーザーに対し、`退勤（自動補正）` 行を追記する。 |
| `AUTO_FIX_HOURS=0` のとき | 補正の退勤時刻は **出勤と同一時刻**（在室0時間扱い）。未退勤の行き止まりを防ぐ用途。 |
| フェーズ1で必要か | **必須ではない。** トリガーを付けなければ自動補正は走らず、**手動で退勤打刻する運用なら記録蓄積だけでもよい。** |
| 付け方 | エディタで `installDailyBatchTrigger()` を**1回実行**（既に同関数のトリガーがあれば二重に作られない）。 |

**まとめ**: 「記録が溜まればよい」だけなら、**Web アプリのデプロイと `doPost` 経路さえ通ればよい**。日次バッチは運用方針が固まってからでよい。

### 1-4. バックエンド・フロント（フェーズ1の確認観点）

- **本番相当**: `GET /api/healthz`、`POST /api/scan` が成功し、シートに行が増える。
- **拒否の確認**: `user_master` に無い ID は `UNKNOWN_USER` で弾かれる（意図どおり）。
- フロントは Vite プロキシ等で `/api` がバックエンドに届くこと。

## フェーズ2 — 記録の品質・運用補助（分析前でも使える）

分析ダッシュボードは後回しでも、次は「ログが崩れにくくする」ための GAS **手動実行**が中心です。

| GAS 関数 | 実行方法 | 用途 |
|----------|----------|------|
| `runAutoFixBatch` | エディタから手動実行可 | 前日分の未退勤をまとめて補正したいとき（トリガー無し運用なら、週1などで手動でも可）。 |
| `installDailyBatchTrigger` / トリガー画面 | 1回設定 | 上記を毎日自動化する。 |
| `rebuildSessionLog` | 手動実行 | `attendance_log` から `session_log` を**再構築**（1行=入退室セッション）。**フェーズ3の集計の前提**になるが、スプシだけで生ログを見るだけなら未実行でもよい。 |

**このフェーズでは**、`doGet`（`mode=users` / `mode=analytics_semester`）は、フロントの「ユーザー一覧・分析表示」やバックエンドの `GET /api/view/...` を使う場合に意味があります。記録だけなら必須ではありません（ただし本番の **ユーザー照合** は `user_master` 読み取りのため `doGet` がデプロイに含まれている必要があります）。

## フェーズ3 — 分析・可視化（後フェーズ）

次の作業は、運用が回ってからでよいものです。

| 対象 | 内容 |
|------|------|
| GAS `setupAnalyticsSheets` | `summary_semester` / `dashboard` 草案や数式の一括作成。`session_log` と連動する前提。 |
| シート `summary_weekly` など | 週次正規表・グラフ（別途設計どおり）。 |
| GAS `doGet` `mode=analytics_semester` | 前期・週平均などを API で返す。 |
| バックエンド `GET /api/view/analytics/semester` | 上記をフロント向けに中継。 |
| フロント「分析」タブ | バーグラフ等のリッチ化。 |

**依存関係のざっくり**: 意味のある週平均表示には、`session_log` の整備 → `summary_*` の式 or GAS → `doGet` の順で揃えることが多いです。

## 附録A. 定期実行の「設定チェックリスト」（必要になったら）

- [ ] スクリプトプロパティ: `GAS_SHARED_SECRET`, `AUTO_FIX_HOURS` が意図どおり。
- [ ] ウェブアプリを**新しいバージョンとして再デプロイ**した（`doPost` / `doGet` を追加したあとに必要になりがち）。
- [ ] トリガー一覧に `runAutoFixBatch` が1本ある（毎日 5 時）。不要なら削除してよい。
- [ ] `user_master` と QR の `user_id` が一致している。
