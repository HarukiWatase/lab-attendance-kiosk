# 研究室勤怠システム
## プレゼン・利用説明（スライド）

**対象:** 研究室メンバー（学生・教員）  
**目的:** 打刻の仕方とデータの流れを共有する

---

## この資料で説明すること

1. **何のためのシステムか**（目的）
2. **学生がキオスクでやること**（打刻・注意点）
3. **画面の「分析」タブ**（補助的な週次表示）
4. **データがどこに記録されるか**
5. **教員・管理者向け**（スプレッドシート側）
6. **困ったとき**・セキュリティの注意

> **スライドの出し方:** [Marp for VS Code](https://marketplace.visualstudio.com/items?itemName=marp-team.marp-vscode) 等で本ファイルを開き、PDF / HTML で出力できます。

---

## システムの目的

- 研究室への **入室・退室を QR（ユーザーID）で記録**する
- 記録は **Google スプレッドシート** に蓄積し、後から集計・報告に使う
- **キオスク（Raspberry Pi + ブラウザ + スキャナ）** を研究室に置いて運用する想定

**正式な月次レポートや教授向けダッシュボード**は、主に **スプレッドシート側** で整える前提です（フロントの分析は補助）。

---

## 全体像（ざっくり）

| 部品 | 役割 |
|------|------|
| **フロント（ブラウザ）** | ID の入力（スキャン）→ API へ送信 |
| **バックエンド（FastAPI）** | 入力チェック・短時間の連打防止 → GAS へ転送 |
| **GAS（Web アプリ）** | スプレッドシートに **1 行ずつ追記** |
| **スプレッドシート** | **`attendance_log`** などにログ保管 |

データは **研究室の PC（Pi）からインターネット経由で GAS** に届きます。

---

## 学生向け：打刻のしかた（キオスク）

1. 画面の **「打刻」** を選ぶ（初期表示）
2. **自分の QR コード** をスキャナにかざす  
   （または開発時は ID 入力 + 確定）
3. 画面に **成功メッセージ** が出れば記録完了
4. **入室と退室は交互**（連続で同じ種別にならないよう API が管理）

**スキャナ設定:** 読取後に **改行（Enter 相当）** が送られる前提です（仕様で CR+LF 想定）。

---

## 注意してほしいこと（打刻）

- **未登録のユーザー ID** は拒否されます（`user_master` に載っている ID のみ）
- **短時間に続けて打てない**時間帯があります（**クールダウン**）。画面の案内に従ってください
- **ネットワークや GAS が不通**のときはエラー表示になります。無理に連打せず、担当者へ

---

## 「分析」タブについて（補助表示）

- **今週（月曜始まりの暦週）** の在室時間の目安を一覧表示します
- **週 15 時間**を目安としてバーを表示（**教授向け月次の判定とは別**の補助指標）
- タブを開いたときに **データを取り直します**
- しばらくすると **自動で「打刻」に戻る**動きがあります（キオスク想定）

---

## データはどこに残るか

- 主な生ログ: スプレッドシートの **`attendance_log`**
  - 例: 日時、ユーザー ID、出勤 / 退勤、リクエスト ID など
- ユーザー名の対応: **`user_master`**（本番ではここに無い ID は打刻不可）

**自分の記録を詳しく見たい**場合は、運用ルールに従い **教員・管理者経由**でスプレッドシート閲覧となることが多いです。

---

## 教員・管理者向け（ざっくり）

- **運用の入口:** `operations-phased-guide.md`（フェーズ別の設定）
- **教授向けダッシュボード・シート操作:** `professor-dashboard-spec-and-manual.md`
- **GAS のコード:** リポジトリの `gas/attendance.gs`（スプレッドシートにデプロイしたものが実体）

月次締めや自動補正などは **GAS の実行・トリガー**で回す設計です（詳細は上記マニュアル）。

---

## 研究室の Raspberry Pi で動かすとき

- **手順のまとめ:** リポジトリの **`deploy/raspberry-pi/README.md`**
- ざっくり:
  - フロントを **`vite build`** して静的ファイル化
  - **nginx** で画面配信 + **`/api` をバックエンドへ**
  - **systemd** で FastAPI（uvicorn）を常駐
  - **Chromium キオスク**で起動時に `http://127.0.0.1/` を開く

初回セットアップ後は **電源投入 → 自動起動** が目標です。

---

## トラブル時（学生・利用者）

| 状況 | まず |
|------|------|
| ずっとエラー | 画面を写真に撮り、**時刻**とともに担当者へ |
| 打刻したのに反映が不安 | **担当者がスプシで `attendance_log` を確認** |
| 画面が真っ暗 / 固まる | Pi の再起動は **担当者判断**（電源・ネットワークも確認） |

**API が生きているか:** 担当者向けに `GET /api/healthz`（`{"ok":true}`）の確認手順が README にあります。

---

## セキュリティ・取り扱い

- **`.env`・GAS の共有秘密**は **リポジトリやスライドに貼らない**
- スプレッドシートの **共有範囲**は研究室ポリシーに従う
- キオスク画面に **個人を特定しうる情報**が出る場合は、撮影・SNS投稿に注意

---

## まとめ

- **学生:** QR で打刻 → 成功表示を確認。連打不可時間・未登録 ID に注意
- **分析タブ:** 今週の目安（補助）。正式な評価はスプシ側の運用に従う
- **記録の正:** **`attendance_log`** と **`user_master`**
- **Pi 運用:** **`deploy/raspberry-pi/README.md`** が手順の本体
