# 研究室勤怠システム

## プレゼン・利用説明

**対象:** 学生・教員（リポジトリや開発ドキュメントは見ない前提）  
**本資料だけ**で概要・使い方・データの流れ・技術構成を把握できます。

---

## この資料の内容

1. **目的**と**データの流れ**
2. **技術スタック**（フロント／API／クラウド／キオスク）
3. **学生向け** 打刻の手順と注意
4. **分析タブ**（画面の補助表示）
5. **スプレッドシート**に何が残るか
6. **教員・管理者**が触る範囲（シート・自動処理）
7. **Raspberry Pi** の起動の考え方
8. **トラブル時**・セキュリティ

---

## システムの目的

- 研究室の **入室・退室** を、**QR コード（ユーザー ID）** で記録する
- 記録は **Google スプレッドシート** に溜まり、**集計・月次報告・教授向けダッシュボード**の材料になる
- 研究室に置いた **小型 PC（Raspberry Pi）＋画面＋ USB スキャナ** で操作する想定（キオスク）

画面上の「分析」は週次の目安表示であり、**正式な評価ルールはスプレッドシート側の運用**に従います。

---

## 技術スタック一覧

| 層               | 採用技術                                 | 役割（一言）                         |
| ---------------- | ---------------------------------------- | ------------------------------------ |
| **画面（UI）**   | React, TypeScript, Vite, Tailwind CSS    | ブラウザ上の打刻・分析画面           |
| **研究室 API**   | Python, **FastAPI**, **Uvicorn**         | 入力検証・クールダウン・GAS への中継 |
| **クラウド処理** | **Google Apps Script (GAS)**             | スプレッドシート読み書き・定期バッチ |
| **データ置き場** | **Google スプレッドシート**              | ログ・マスタ・集計シート             |
| **キオスク端末** | Raspberry Pi OS, **nginx**, **Chromium** | 画面表示と API の常時起動            |

---

## フロントエンド（画面）の詳細

- **React 18** … UI を部品化して組み立てるライブラリ
- **TypeScript** … JavaScript に型を付け、バグを減らす
- **Vite** … 開発時の高速サーバ・**本番用に静的ファイル（HTML/CSS/JS）をビルド**
- **Tailwind CSS** … ユーティリティ中心のスタイリング

**本番では** ビルド結果を **nginx がそのまま配信**し、ブラウザは **同じアドレスの `/api/...`** だけ API にアクセスします（別ポートを意識しなくてよい構成）。

---

## バックエンド（研究室 API）の詳細

- **FastAPI** … Python のモダンな Web API フレームワーク（OpenAPI ドキュメント自動生成など）
- **Uvicorn** … FastAPI を動かす **ASGI サーバ**（本番では systemd から常駐起動）
- **httpx** … GAS の Web アプリ URL へ **HTTP POST** するときに使用
- **python-dotenv** … 接続先 URL や秘密鍵を **環境変数（.env）** から読み込み
- **Pydantic** … リクエスト JSON の形を検証

**主なエンドポイント（代表）**

- `POST /api/scan` … 打刻（スキャナ／本番）
- `GET /api/healthz` … 生存確認（`{"ok":true}`）
- `GET /api/view/users` … 登録ユーザー一覧（表示名など）
- `GET /api/view/analytics/week-calendar` … 分析タブ用の週次データ  
  開発用に `POST /api/mock-scan` がある場合があります。

---

## GAS・スプレッドシートの詳細

- **Google Apps Script** をスプレッドシートに紐づけ、**ウェブアプリとして公開**
- 研究室 API から **POST** されると **`doPost`** が動き、**`attendance_log` シートに 1 行追加**
- 前回の状態に応じて **出勤／退勤を自動トグル**（どちらを記録したかは API 経由で画面にも返る）
- **`doGet`** でマスタ読み取りなどに応答（ユーザー照合・分析用データの取得）

**シート例（運用で使う名前）**

- **`attendance_log`** … 打刻の生ログ（日時・ID・出勤/退勤・リクエスト ID 等）
- **`user_master`** … 利用可能な ID と表示名・有効フラグ（**ここに無い ID は本番では打刻不可**）
- **`session_log`** など … 入退室をまとめたログ（GAS で再構築する運用）
- **教授向けダッシュボード用シート** … 月次集計・グラフ（運用で整備）

---

## キオスク（Raspberry Pi）側の詳細

- **OS:** Raspberry Pi OS（Debian 系、例: Bookworm）
- **nginx** … フロントの静的ファイルを配信し、パス **`/api` 以降だけ** 裏の FastAPI（例: `127.0.0.1:8000`）へ **リバースプロキシ**
- **systemd** … `uvicorn` で FastAPI を **起動時自動起動・落ちたら再起動**
- **Chromium キオスクモード** … 電源投入後、**全画面で `http://127.0.0.1/`** を開く（ログイン後の自動起動設定と組み合わせることが多い）

**USB QR スキャナ**はキーボードと同様に動き、**読取後に Enter（改行）** が送られる設定を推奨します。

---

## データの流れ（打刻 1 回）

```
[スキャナ] → 画面（React）
    → POST /api/scan（FastAPI）
        → ユーザー照合・クールダウン判定
        → POST（GAS ウェブアプリ）
            → スプレッドシートに 1 行追記
    ← 結果 JSON ← GAS 応答 ← FastAPI ← 画面表示
```

**インターネットに出るのは主に GAS への通信**です（Google のサーバ）。

---

## 学生向け：打刻の手順

1. 画面で **「打刻」** を選ぶ（最初からそうなっていることが多い）
2. **自分の QR** をスキャナにかざす（**読み取り後に改行が入る**設定）
3. **成功メッセージ**が出たらその回の記録は完了
4. **入室と退勤は交互**（システムが前回の記録から次を決める）

※ 開発用 PC では ID を手入力してボタンを押すこともありますが、**研究室の本番機では入力欄が隠れる**ことがあります。

---

## 注意（学生向け）

- **研究室に登録されていない ID** はエラーになります（マスタに載せてもらう必要あり）
- **すぐに続けて同じ人が打てない**時間があります（**クールダウン**・誤読取り対策）
- **Wi-Fi や Google 側の不調**で一時的に失敗することがあります。**連打せず**担当者へ

---

## 「分析」タブ（補助表示）

- **今週（月曜始まりの暦週）** の在室時間の合計を、メンバーごとに表示
- **週 15 時間**を目安にバーを表示（**研究室の月次ルールとは別物**である旨、画面にも注記あり）
- タブを開いたタイミングで **データを取り直す**
- **約 60 秒で「打刻」に戻る**動きがあり、長居しにくくしています

---

## スプレッドシートに残るもの（利用者が知る範囲）

**`attendance_log`（打刻のログ）**

- いつ・誰が・**出勤か退勤か**・どの経路から（スキャナ等）・重複防止用の ID など

**`user_master`（名簿）**

- 打刻を許可する **ユーザー ID** と **表示名**、利用可否

詳細な列は運用担当が管理します。**自分の履歴を見たい**場合は、研究室ルールに従い教員・担当者へ相談してください。

---

## 教員・管理者向け（リポジトリ不要で知っておくこと）

- **正本はスプレッドシート**です。異常時は **`attendance_log`** をまず確認します。
- **教授向けの月次ダッシュボード**は、同じブック内の専用シートや GAS で更新される設計です（**締め日・共有のルールは研究室で決める**）。
- **前日の退勤忘れ**などを機械的に埋める **日次バッチ**（自動補正の行を追記）を、GAS の **時間トリガー**で回す運用があります。使う／使わないは方針次第です。
- **セッション一覧**（`session_log`）は、ログから GAS で **再計算・再構築**する処理があり、分析の前提になります。

※ 具体的なボタン名・実行手順は **運用担当者向けの手順**に委ねます。

---

## Raspberry Pi：起動のイメージ（初回セットアップ後）

1. **電源 ON** → OS 起動 → 可能なら **デスクトップ自動ログイン**
2. **nginx** が動き、`http://127.0.0.1/` で画面（ビルド済みフロント）が開く
3. **systemd** が **Uvicorn（FastAPI）** を裏で起動し、`/api` が nginx 経由で届く
4. **Chromium キオスク**が同じ URL を全画面表示

**更新したとき（担当者）** の典型手順: 最新コードを取得 → フロントを再ビルド → API サービス再起動 → nginx 設定に問題なければ再読み込み。

---

## トラブル時

| 状況               | 利用者                 | 担当者の目安                                       |
| ------------------ | ---------------------- | -------------------------------------------------- |
| 画面がエラーのまま | 時刻と画面を記録し報告 | `attendance_log`・ネットワーク・GAS の稼働を確認   |
| 打った気がない     | 連打しない             | ログの時刻・`request_id` で追跡                    |
| 真っ暗・固まる     | 担当者へ               | 電源・HDMI・自動起動・手動で Chromium / API 再起動 |

**API が応答するか**（担当者）: 同じ Pi 上でブラウザまたはコマンドから **`http://127.0.0.1/api/healthz`** にアクセスし、**`{"ok":true}`** なら API 経路は生きています。

---

## セキュリティ・取り扱い

- **API と GAS の共有秘密・環境設定ファイル**は **公開リポジトリ・スライド・チャットに貼らない**
- スプレッドシートの **共有リンクの範囲**（大学アカウントのみ等）は研究室ポリシーに合わせる
- キオスク画面に **氏名が出る**ため、**無断撮影・SNS投稿**に注意

---

## まとめ

- **学生:** QR で打刻、メッセージ確認。**未登録 ID・クールダウン**に注意
- **分析タブ:** 今週の目安のみ。**評価の正はスプレッドシート運用**
- **技術:** React/TS/Vite 画面 → FastAPI/Uvicorn → GAS → スプレッドシート；Pi 上は nginx ＋ systemd ＋ Chromium
- **困ったら:** 記録して担当者へ。**正本はスプレッドシートのログ**
